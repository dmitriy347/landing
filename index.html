ОСНОВНОЙ КЕЙС

tool-issuance-bot

ПРОБЛЕМА:

Бухгалтеры каждый месяц вручную переносят данные из 1С в Word, составляют акты приёма-передачи инвентаря для каждого сотрудника. Это отнимает время + вероятность допустить ошибки при копировании.

РЕШЕНИЕ:

Telegram-бот как интерфейс, FastAPI как backend, AI - звено пайплайна. Бухгалтер отправляет боту скриншот окна 1С → Groq Vision API распознаёт ФИО сотрудника(ов) → backend ищет сотрудника в БД и подбирает нужный шаблон акта → docxtpl генерирует готовый документ → бот мгновенно возвращает DOCX.

01скрин 1С
→
02Groq Vision API
→
03поиск в БД
→
04генерация DOCX
→
05доставка документа

РЕЗУЛЬТАТ:

  • рабочий end-to-end pipeline: скриншот → AI извлекает данные → БД → готовый документ;
  • чистое разделение слоёв: бот (транспортный слой) / FastAPI API / services (ai_extractor, excel_parser, docx_generator);
  • полностью асинхронный пайплайн;
  • сотрудники и инвентарь связаны не по ID, а через сравнение строк ФИО, т.к. данные приходят из 1С;
  • единая обработка ошибок через FastAPI HTTPException;
  • покрытие тестами AI-экстрактора, Excel-парсеров и генератора документов (pytest, async).

ДОПОЛНИТЕЛЬНЫЙ КЕЙС

SkiTrip Journal

Production веб-приложение - журнал поездок на горнолыжные курорты (Django + DRF), довел от прототипа до деплоя.

КЛЮЧЕВЫЕ МОМЕНТЫ:

  • REST API на 17 эндпоинтов, JWT-аутентификация, rate limiting через DRF Throttling + Redis;
  • асинхронная генерация миниатюр через Celery + Redis;
  • тестовое покрытие >90% (pytest);
  • CI/CD на GitHub Actions (lint → test → build → deploy), Docker, продакшен на VPS (Nginx + Gunicorn + SSL).

Демо временно не развёрнуто (сервер отключён), но проект полностью поднимается локально командой: docker-compose up. CI/CD pipeline и тестовое покрытие - в репозитории.
